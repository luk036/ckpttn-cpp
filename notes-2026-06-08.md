# Session Notes — 2026-06-08

## Overview

Major refactoring of the MidLvl partition managers to use `FMBiGainCalc`/`FMKWayGainCalc` for gain computation instead of manual delta calculation. Benchmark comparison of FM-only vs Multilevel (ML) partitioning across 3 testcases and 3 K-values. Fixed BPQueue out-of-bounds crash for K>2.

---

## 1. MidLvlPartMgr Refactoring (2-way exhaustive)

### Problem
`MidLvlPartMgr::optimize()` manually computed delta cost by scanning all nets per flip in the HamCycle visitor. This duplicated the gain calculator logic and did not use `FMBiGainCalc`.

### Solution
- **Removed** manual delta computation from the visitor
- **Added** `FMBiGainCalc<Gnl>` member (not `FMBiGainMgr` — no FM buckets/locking/waiting lists)
- Gains initialized from `gain_calc.init(part)` → copied to local `current_gain[]` vector
- Per-flip neighbor gains updated by calling `gain_calc.update_move_*_net()` publicly
- `current_gain[v] = -gain` after each move (reverse move gain)
- `FMBiConstrMgr` tracks weight balance; snapshot only when `LegalCheck::AllSatisfied`

### Files
- `include/ckpttn/MidLvlPartMgr.hpp` — Uses `FMBiGainCalc<Gnl>` instead of `FMBiGainMgr<Gnl>`
- `source/MidLvlPartMgr.cpp` — Rewritten: gain calc init + manual gain tracking + balance snapshot
- `include/ckpttn/FMBiGainCalc.hpp` — Added `get_init_gain_list()` public accessor
- `test/source/test_MidLvlPartMgr.cpp` — Added `bal_tol` to all constructors

## 3. FMKWayGainMgr Crash Fix (K>4)

### Problem
K=5 on ibm01 crashed with SIGSEGV (heap corruption). Root cause: `FMGainMgr` constructor created BPQueue with range `[-pmax, +pmax]`, but after `update_move` neighbor deltas, gain values reached up to `±(K-1)·pmax`, causing out-of-bounds bucket array access.

### Fix
Changed BPQueue range from `[-pmax, +pmax]` to `[-(K-1)·pmax, +(K-1)·pmax]` in `FMGainMgr` constructor.

```cpp
// Before (source/FMGainMgr.cpp:31)
this->gain_bucket.emplace_back(BPQueue<node_t>(-pmax, pmax));
// After
const auto range = static_cast<int>(this->num_parts - 1) * pmax;
this->gain_bucket.emplace_back(BPQueue<node_t>(-range, range));
```

The Python `FMGainMgr` already had this bound: `bound = self.pmax * (num_parts - 1)`.

### Files
- `source/FMGainMgr.cpp` — BPQueue range fix

### Max-Iteration Guard
Added `for (int iter = 0; iter < 100; ++iter)` guard in `PartMgrBase::optimize()` to prevent pathological non-convergence in K-way FM (seed=46 with K=5 on ibm01 triggered >1000 passes).

- `source/PartMgrBase.cpp` — max-iterations guard

## 4. Benchmark: FM-only vs MLPartMgr

### Setup
- **Testcases**: ibm01 (12,752 modules), p1 (833 modules)
- **K-values**: 2, 3, 5
- **Parameters**: `bal_tol=0.45`, `limitsize=10`, `runs=5` (seeds 47-51)
- **Methods**: FM-only (`FMPartMgr::legalize + optimize`) vs ML (`MLPartMgr::run_Partition`)

### Results

| Testcase | K | FM avg | ML avg | ML impr | FM time | ML time |
|----------|---|--------|--------|---------|---------|---------|
| ibm01 | 2 | 675.2 | 486.0 | **+28.0%** | 0.34s | 0.47s |
| ibm01 | 3 | 2976 | 1475.6 | **+50.4%** | 1.67s | 2.68s |
| ibm01 | 5 | 3712.2 | 2023.2 | **+45.5%** | 5.21s | 21.2s |
| p1 | 2 | 98.6 | 72.4 | **+26.6%** | 6.7ms | 18.3ms |
| p1 | 3 | 169.2 | 114.6 | **+32.3%** | 132ms | 127ms |
| p1 | 5 | 258.8 | 174.6 | **+32.5%** | 256ms | 531ms |

### Key Findings
1. **ML always beats FM** (26-50% cost improvement)
2. **K-way** benefits more from multilevel than 2-way (45-50% vs 26-28%)
3. **Runtime tradeoff**: ML is 1.4-4× slower but quality gain scales with problem size
4. **Parity confirmed**: ML without contraction equals FM-only (verified by test)
5. **Limitsize=10** and **limitsize=2000** produce identical costs — the contraction ratio guard `n*3/2 < original` drives depth, not the limit

### Benchmark Infrastructure
- `test/source/test_benchmark_ibm01.cpp` — Consolidated benchmark test
- Script compares FM-only and ML on the same initial partition (exact same RNG seed)

## 5. MLMidLvl Assessment

MLMidLvlPartMgr and MLMidLvlKWayPartMgr produce **identical results** to MLPartMgr on all benchmarks. The exhaustive MidLvl leaf (<25 modules) is never reached on graphs >25 modules, so both use the same contraction + FM optimize pipeline.
